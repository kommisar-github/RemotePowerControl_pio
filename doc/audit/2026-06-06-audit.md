### Severity-calibration notes (the higher-tier verifier's job)

- **C-1 — "crash" overstated, defect real.** PubSubClient does not null-terminate but its receive buffer is `MQTT_MAX_PACKET_SIZE` (1024 B), so reading index 2 of a short payload reads *stale buffer bytes*, not unmapped memory — a hard crash is unlikely. The real, confirmed risks are (a) reading uninitialized/stale content (undefined behavior) and (b) **bounded mis-actuation**: the downstream guard `switch_idx > -1 && switch_idx < 3` (`main.cpp:500`) plus the action-char check (`:502`–`:507`) mean a wrong relay fires only if the stale `payload[2]` happens to land in `'1'..'3'` *and* `payload[0]` in `{'0','1','3'}`. Still a memory-safety defect on the command path; the fix is a one-line `if (length < 3) return;`. **Kept P0** for cost/benefit (trivial fix, command-path correctness), not for crash risk.
- **C1 — DOWNGRADE CRITICAL→MEDIUM.** The `{stat1/2/3}` placeholders are overwritten by the 2 s `getStatus()` JS poll (`strings1.h:7`, `document.getElementById('st1').innerHTML=resp.devs[0].stat`), so the status text self-corrects within 2 s. Only the **device-header labels** (`{DEVICE1}`, `{device2}`, `{device3}`) render as literal placeholder text permanently. The toggle controls use hard-coded indices `send_cmd(this,0..2,…)` and are fully functional. `{ver}` (`TAG_VER`) does not even appear in `PAGE_BUTTONS`, so "version never substituted" is really "version field is dead" (corroborates web L4). **No functional, safety, or data impact → not CRITICAL.**
- **F-01 — confirmed worst-case.** Because the relay is **momentary** (it simulates a front-panel power-button press), the boot-time HIGH glitch on GPIO5 is not a benign steady state — it injects a *simulated power-button press to the attached host on channel 2 on every ESP32 reset/flash*. That can power the controlled machine on or off unexpectedly. This is the single highest-blast-radius defect on the ESP32 target.

## 2. Invariant verdicts

### 2.1 Secrets — **PASS (CLEAN)**
All credentials flow `sysenv.SMARTHOME_*` → `platformio.ini -D` macros → `main.cpp` macro consumption, with **no source-side literal and no fallback `#define`**:
- `main.cpp:54-60` — `SSID=_WIFI_SSID_`, `PWD=_WIFI_PWD_`, `OTA_PASSWD=_OTA_PASSWD_`, `mqtt_server=_MQTT_BROKER_`.
- `platformio.ini:19-26,42` — `-D _WIFI_SSID_=\"${sysenv.SMARTHOME_WIFI_SSID}\"` etc.; `ota_pass = ${sysenv.SMARTHOME_OTA_PASSWD}`.
- Grep for `#define _WIFI_SSID_|_WIFI_PWD_|_MQTT_BROKER_|_OTA_PASSWD_` in `src/` → **no matches** (no hard-coded fallback). Verdict holds for source; git-history scan is out of this audit's scope (recommend a one-time history secret-scan as a separate task).

### 2.2 Boot-safe-GPIO — **FAIL on ESP32 / PASS on ESP8266**
With the relay drive confirmed **active-HIGH**:
- **ESP32 — FAIL.** `POWER_SWITCH2_GPIO=5` (`Application.h:45`). GPIO5 has an internal pull-up asserted at reset and is driven during the boot window before `setup()` runs `pinMode/digitalWrite LOW`. Active-high relay → **fires a momentary press on every reset/flash**. (F-01.)
- **ESP8266 — PASS (but document).** `POWER_SWITCH2_GPIO=D8` = GPIO15 (`Application.h:33`). GPIO15 requires an external pulldown and is **held LOW at boot**, so an active-high relay stays *off* through reset — boot-safe for this design. The residual concern (F-02) is the inverse — a relay board that pulls GPIO15 high could block boot — and the misleading source comment. **Not a relay-firing risk here**; downgraded to a documentation/boot-reliability item.
- **Cross-MCU trap to record:** GPIO number "5" is **D1 = boot-safe on ESP8266** but a **strapping pin on ESP32**. The pin map almost certainly inherited the ESP8266 reasoning onto the ESP32 column. Fix: reassign ESP32 SWITCH2 to a non-strapping output (e.g. GPIO13/14/27) and confirm polarity with `/power`; capture the rationale in `HARDWARE.md`.

### 2.3 Non-blocking-loop — **PASS (one backlog caveat)**
- Grandfathered blocking unchanged and is the only *intentional* blocking on the `loop()` path: `RSwitch::short_push()` `delay(800)` (`RSwitch.cpp:116`) and `long_push()` `delay(10000)` (`:124`).
- **Caveat (I-1, not a regression):** `MqttClient::reconnect()` calls the synchronous `_client->connect()` (`MqttClient.cpp:152/155`), which blocks up to the PubSubClient/WiFiClient socket timeout. It is **bounded** (socket timeout, not unbounded), **WiFi-gated** (bails immediately if `WiFi.status()!=WL_CONNECTED`, `:144`), and **rate-limited** to one attempt per reconnect interval (`:130`). It only stalls when WiFi is up but the broker is unreachable. This is the standard synchronous-PubSubClient pattern and **pre-existing** — it does not introduce *new* blocking, so the invariant is not violated, but it belongs on the non-blocking backlog alongside the relay pulses.

## 4. Systemic theme — **VALIDATED: "silently-ignored configuration"**

This is a real, recurring failure *class*, not a coincidence of point bugs. Confirmed instances:
- **Broken preprocessor guards** — `#ifdef ESP8266 && !defined(NO_GLOBAL_MDNS)` (`OTA.cpp:13,16`; also `:67` per net H1). `#ifdef` consumes one identifier; the conditions are silently ignored.
- **Overloads that drop arguments** — `MqttClient::begin(server,id,enabled)` forces `_enabled=true` (mqtt M-1); `WiFiConnect::begin(…,mdns_enabled)` drops the flag (net M2). The caller's explicit config is silently discarded.
- **String substitution that silently no-ops** — web C1 `replace("device1",…)` leaves most placeholders intact.
- **Documented-but-contradicted hardware intent** — boot-safe comment (`Application.h:29`) vs strapping-pin assignments (F-01/F-02).

**Recommendation (pattern-level, for the future fix phase):** adopt a "config must be honored or rejected, never silently dropped" rule — (a) replace every `#ifdef X && …` with `#if defined(X) && …` and add a CI `-Wundef`/`-Werror` style guard; (b) audit all multi-arg `begin()` overloads so a passed flag is either used or removed from the signature; (c) prefer whole-token substitution/templating for the web page. This is durable guidance — recommend it be routed to GUIDELINES via the next review-gated consolidation wave (not written now).

## 6. Prioritized remediation order (for a future fix phase — NOT executed now)

Ranked by severity × blast-radius × invariant impact.

### P0 — must fix before any release build
1. **F-01 — ESP32 SWITCH2 off strapping pin GPIO5.** Reassign to a non-strapping output (GPIO13/14/27), confirm polarity with `/power`, document in `HARDWARE.md`. *Invariant: boot-safe-GPIO. Blast radius: every ESP32 boot actuates the host.* (`Application.h:45`)
2. **BUG-01 — `longPress()` dead branch.** 2nd condition → `isPowerOn()` (or `!isPowerOff()`), so force-off sets `POWERING_OFF` not `UNDEFINED`. *Contract C2/C3; force is a primary command.* (`RSwitch.cpp:308`)
3. **C-1 — MQTT payload length guard.** Add `if (length < 3) return;` at the top of `mqtt_received_callback`. *Memory safety on the command path; one line.* (`main.cpp:496`)

### P1 — should fix (silent failures / reconnect correctness / build risk)
4. **net H2 — reset `mqtt_conn_timestamp` in the WiFi-lost branch** (`Test.cpp:188-196`). Fixes the WIFI_RESTORED→MQTT_LOST mis-ordering (net M3) so `EVENT_MQTT_RESTORED` fires and `refresh_all_switches_state()` runs → HA recovers after reconnect. *Contract C4; high functional blast radius on every reconnect.*
5. **net H1 — `#ifdef X && …` → `#if defined(X) && …`** at `OTA.cpp:13,16,67` (and audit for siblings). *Latent today; lead instance of the systemic theme.*
6. **info JSON overflow {M-2,M-3,F-07}** — bump `DynamicJsonDocument`/`BUF` to 2048 **or** split the `connection_info=true` frame; check `serializeJson`/`publish` return. *Silent telemetry loss to HA.* (`main.cpp:87,442`)

8. **firmware F-03 / F-04 — `Log` macro decl/def mismatch** (`LOG_DDEBUG` vs `LOG_DEBUG`; `LOG_ERROR` const-qualifier). *Linker/overload error the moment the macro is used; reconcile header and impl.* (`Log.h:18,21` vs `Log.cpp:68,78`)
9. **firmware F-13 — `#include "log.h"` vs file `Log.h`.** Case-only mismatch builds on Windows but **breaks any case-sensitive (Linux) CI**. Promote to P0 *if* Linux CI is on the roadmap; otherwise P1. (`Application.h:13` and all sibling includes)
